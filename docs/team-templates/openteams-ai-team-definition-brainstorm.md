# OpenTeams AI 团队定义头脑风暴

## 目标

当前 OpenTeams 已经具备“成员 + 团队协议”的基础，其中成员由职责定义、Skill 和 MCP 能力定义。要让一个 AI 团队真正能干活，团队模板还需要描述：它为什么存在、能接什么活、谁有决策权、如何交接、如何验收、出了问题如何升级，以及运行时有哪些边界。

## 一个团队需要定义的内容

### 1. 团队身份

- 团队名称：例如“全栈产品交付小队”
- 团队目标：这个团队主要解决什么问题
- 适用任务类型：feature、bugfix、review、refactor、research、release 等
- 不适用范围：明确不能承接或不建议承接的任务
- 默认工作模式：chat mode、workflow mode，或两者都支持
- 默认交付物：代码、设计说明、测试报告、PR、文档、迁移方案等

### 2. 成员定义

每个成员不应只定义“职责 + skill + mcp”，还需要补充以下内容：

- 成员身份：名称、角色、默认 runner/model、是否可被 @mention
- 职责边界：负责什么，不负责什么
- 决策权限：哪些问题可独立决定，哪些必须请用户或 lead 确认
- 输入契约：该成员启动工作前需要哪些信息
- 输出契约：该成员完成后必须产出什么格式的内容
- 交接对象：完成后默认交给谁继续处理
- 领域知识：业务知识、技术栈知识、项目规范、质量标准
- Skill 能力：可调用的可复用工作流或专业能力
- MCP 能力：可访问的工具、系统、外部资源和自动化能力
- 文件/命令权限：可读写路径、可运行命令、是否需要审批
- 质量责任：该成员对哪些质量门禁负责
- 升级条件：遇到哪些情况必须停止并请求协助

### 3. 团队协议

团队协议是让多个 AI 成员协作的“操作系统”。建议定义：

- 任务入口：用户输入如何转化为团队任务
- 分工规则：谁先看、谁拆解、谁执行、谁复核
- Lead 机制：是否有 lead；lead 是协调者、计划生成者还是最终审核者
- @mention 路由：什么情况下自动或建议 @mention 某个成员
- 交接格式：成员之间交接时必须包含上下文、产出、风险和下一步
- 冲突处理：设计、前端、后端、QA 意见不一致时谁裁决
- 并发规则：哪些任务可并行，哪些任务必须串行
- 阻塞规则：成员等待输入、等待审批、等待其他成员时如何汇报
- 复核规则：哪些产出必须被另一个成员 review
- 完成标准：团队整体 Definition of Done

### 4. 上下文与知识配置

真正可工作的团队需要清楚它能读取哪些知识，以及如何保持上下文同步。

- 项目知识入口：README、docs、架构文档、API 文档、设计规范
- 代码范围：团队默认关注哪些目录、模块、服务或前端页面
- 业务术语表：领域概念、对象关系、常用缩写
- 历史决策：重要架构决策、产品约束、兼容性要求
- 会话记忆策略：哪些信息写入 shared blackboard，哪些只属于本轮
- 上下文同步策略：成员完成工作后需要同步哪些事实给团队

### 5. 工具与 MCP 配置

MCP 不只是“有工具”，还要定义工具的边界和使用规则。

- 工具名称与用途：浏览器、数据库、GitHub、Linear、Figma、测试运行器等
- 允许成员：哪些成员可以使用该 MCP
- 权限范围：只读、可写、可执行、是否需要审批
- 使用前提：需要哪些环境变量、登录状态、服务地址
- 失败策略：工具不可用时如何降级
- 产出格式：MCP 执行后需要留下什么日志、截图、链接或报告

### 6. 工作流定义

团队模板应定义常见任务的默认流程，而不是每次从零协商。

- feature 流程：需求澄清 -> 设计 -> 技术方案 -> 实现 -> QA -> 验收
- bugfix 流程：复现 -> 定位 -> 修复 -> 回归 -> 记录根因
- review 流程：读取变更 -> 风险检查 -> 测试建议 -> 结论
- refactor 流程：范围界定 -> 风险评估 -> 分步修改 -> 回归验证
- release 流程：变更摘要 -> 检查清单 -> CI/测试 -> 发布结论

### 7. 验收与质量门禁

没有验收标准的团队很难稳定交付。建议定义：

- 团队 Definition of Done
- 每类任务的验收清单
- 必跑检查：typecheck、lint、unit test、integration test、e2e test 等
- 可跳过检查的条件：例如纯文档变更不跑测试
- 质量负责人：谁判断是否可交付
- 证据要求：测试输出、截图、日志、PR 链接、设计对比结果

### 8. 权限、安全与风险控制

AI 团队需要明确“能做什么”和“不能做什么”。

- 文件系统权限：读写路径、禁止路径、生成物存放位置
- 命令权限：允许命令、危险命令、审批命令
- 外部系统权限：GitHub、数据库、部署、工单系统等
- 数据安全：secret、生产数据、用户隐私的处理规则
- 破坏性操作策略：删除、重置、迁移、发布必须审批
- 审计要求：关键操作需要记录原因、命令和结果

### 9. 运行时资源与调度

为了让团队可控运行，需要定义运行时参数。

- 成员并发上限：同一成员是否可同时跑多个任务
- 团队并发策略：哪些成员可以并行执行
- 时间预算：单轮最大时长、长任务汇报频率
- Token/上下文预算：长任务如何压缩和传递上下文
- 重试策略：失败后重试次数、是否换成员、是否需要用户确认
- 暂停/恢复策略：用户中断后如何保存现场

### 10. 观测与结果沉淀

团队需要把过程和结果沉淀下来，便于复盘和复用。

- 工作记录：谁做了什么、改了哪些文件、跑了哪些检查
- 决策记录：关键取舍、被拒绝的方案、风险接受原因
- 产物索引：文档、截图、测试报告、PR、设计稿链接
- 质量指标：完成率、返工率、失败原因、平均交付时间
- 模板反馈：哪些成员定义不足、哪些 MCP 缺失、哪些流程需要调整

## 最小可用版本

如果先做产品 MVP，建议团队模板至少包含：

1. 团队名称、目标、适用任务类型。
2. 成员列表：职责、领域知识、Skill、MCP、输入输出契约。
3. 团队协议：分工规则、交接格式、完成标准、阻塞处理。
4. 权限与工具：每个成员可用 MCP、可操作范围、审批要求。
5. 质量门禁：哪些检查必须跑，谁负责最终质量结论。

## 关键结论

一个能真正干活的 AI 团队，不只是多个 agent 的集合，而是“角色能力 + 协作协议 + 上下文边界 + 工具权限 + 质量门禁 + 运行时调度”的组合。成员定义解决“谁能做什么”，团队定义解决“什么时候谁来做、做到什么程度、如何证明完成”。这两层都明确后，OpenTeams 的团队模板才具备可复用、可调度、可审计的基础。
